# Core Data

## Description

Core Data is Apple's framework for managing the model layer of iOS, macOS, watchOS, and tvOS applications. It is not an ORM in the traditional sense — it is an **object graph manager with persistence capabilities**. Core Data sits between your in-memory object model and the persistent store (typically SQLite, but also XML, binary, and in-memory stores).

Core architecture:
- **NSPersistentContainer** — the top-level object that manages the entire stack. Introduced in iOS 10, it simplifies setup by bundling the managed object model, persistent store coordinator, and a default managed object context.
- **NSManagedObjectModel** — the schema definition (entities, attributes, relationships) loaded from a `.xcdatamodeld` file or created programmatically.
- **NSPersistentStoreCoordinator** — mediates between the managed object contexts and the underlying store files. Handles store type selection (SQLite, binary, in-memory, XML) and migration.
- **NSManagedObjectContext** — a "scratch pad" for managed objects. Contexts are the primary working interface — you fetch, insert, delete, and modify objects within a context. Contexts can be nested (a child context writes to its parent).
- **NSManagedObject** — the base class for entity instances. Properties are dynamically resolved at runtime using Key-Value Coding (KVC). NSManagedObject subclasses can be generated by Xcode.
- **NSFetchedResultsController** — a controller that executes a fetch request and monitors for changes, delivering updates to a `UITableView` or `UICollectionView` with efficient diffing and animation.
- **Faulting** — Core Data's lazy loading mechanism. Managed objects are initially faults (placeholders) and only load their data from the store when a property is accessed. This keeps memory usage low for large datasets.

**Under the hood:** Core Data's SQLite store is not a general-purpose SQLite database. It uses a proprietary schema with metadata tables, Z_PRIMARYKEY, Z_METADATA, and Z_MODELCACHE. You should not query Core Data's SQLite file directly with raw SQL — the schema is private and changes between OS versions.

## When to Use

- **Medium-complexity local data models** — entities with relationships (one-to-many, many-to-many), validation rules, and inverse relationships. Core Data's object graph management excels when you need to navigate relationships bidirectionally.
- **Undo/redo support** — Core Data has built-in undo management via `NSManagedObjectContext.undoManager`. For document-based apps (text editors, drawing apps), this is a significant advantage over raw SQLite.
- **Incremental saving and change tracking** — Core Data tracks which properties changed and only writes modified columns. This is more efficient than writing entire records.
- **Integration with SwiftUI/UIKit** — `@FetchRequest` in SwiftUI and `NSFetchedResultsController` in UIKit provide seamless data-driven UI updates with minimal boilerplate.
- **Data synchronization with CloudKit** — `NSPersistentCloudKitContainer` provides automatic sync between Core Data and iCloud, handling conflict resolution and offline support.
- **Batch operations on large datasets** — `NSBatchDeleteRequest` and `NSBatchUpdateRequest` perform direct SQLite operations without loading objects into memory, essential for deleting or updating thousands of records.

## When NOT to Use

- **Simple key-value storage** — `UserDefaults` or a JSON file is sufficient for storing app preferences, settings, or small cached responses. Core Data adds significant overhead for this use case.
- **High-throughput write scenarios** — inserting 10,000+ records per second (e.g., sensor data logging, financial tick data). Core Data's change tracking and faulting add overhead. Use raw SQLite via SQLite.swift or GRDB.
- **When your team lacks Core Data experience** — the learning curve is steep. Misusing contexts, not understanding faulting, or skipping migration testing leads to data corruption and crashes in production.
- **When you need fine-grained SQL control** — complex joins, window functions, CTEs, or custom indexes. Core Data's fetch request API does not expose the full power of SQLite. Use GRDB or SQLite.swift.
- **Cross-platform applications** — Core Data is Apple-only. If your app also runs on Android or the web, use a cross-platform solution (Realm, SQLite, or a sync engine like PowerSync/WatermelonDB).
- **When the data model is unstructured or schemaless** — Core Data requires a fixed schema. If your data is highly dynamic (user-defined fields, JSON documents), store it as JSON in SQLite or use a document database.
- **Very small apps with simple CRUD** — the boilerplate of defining entities, generating NSManagedObject subclasses, and setting up the stack is overkill for an app with 2–3 entity types and no relationships.

## Tradeoffs

| Dimension | Core Data | GRDB / SQLite.swift (raw SQLite) |
|-----------|-----------|---------------------------------|
| **Setup complexity** | High (model, container, contexts, migration) | Low (open database, create tables) |
| **Object graph management** | Built-in (relationships, inverse relationships, delete rules) | Manual (joins, foreign keys) |
| **Change tracking** | Automatic | Manual (you track what changed) |
| **Undo/redo** | Built-in | Must implement yourself |
| **Performance (bulk inserts)** | Moderate (change tracking overhead) | High (direct SQL, no overhead) |
| **Memory management** | Faulting keeps memory low | You manage memory explicitly |
| **Migration** | Lightweight migration for simple changes; custom for complex | Manual ALTER TABLE scripts |
| **Thread safety** | Complex (each context is thread-bound) | SQLite is thread-safe with WAL mode |
| **Query flexibility** | NSPredicate (limited compared to SQL) | Full SQL (joins, CTEs, window functions) |
| **SwiftUI integration** | `@FetchRequest` built-in | Manual integration |

| Dimension | Core Data | Realm |
|-----------|-----------|-------|
| **Performance** | Moderate (SQLite-backed, faulting) | High (zero-copy, custom engine) |
| **Memory** | Faults reduce memory but add latency | Live objects always in memory |
| **Schema migration** | Built-in (lightweight + custom) | Built-in (simpler but less flexible) |
| **Cloud sync** | NSPersistentCloudKitContainer (Apple ecosystem) | Atlas Device Sync (MongoDB ecosystem) |
| **Binary size** | ~0 MB (system framework) | ~5–8 MB (bundled library) |
| **Platform support** | Apple only | iOS, Android, React Native, .NET, Node.js |

## Alternatives

- **GRDB.swift** — a robust Swift SQLite wrapper with migrations, full SQL support, and observation (Combine/SwiftUI integration). Best when you need raw SQL power without Core Data's complexity.
- **Realm** — a mobile-first database with a custom storage engine (not SQLite). Faster than Core Data for most operations, simpler API, and cross-platform. Adds ~5–8 MB to your binary.
- **SQLite.swift** — a type-safe Swift wrapper over SQLite. Minimal abstraction. You write SQL directly. Best for developers who want full control.
- **UserDefaults** — for simple key-value data (settings, flags, last-opened timestamps). Not a database but often the right answer for simple persistence.
- **File-based storage** — encode model objects as JSON/Codable and write to disk. Works for small datasets with simple access patterns.
- **SwiftData** (iOS 17+) — Apple's modern replacement for Core Data. Uses Swift macros (`@Model`) for a much simpler API. Under the hood, it still uses Core Data's SQLite stack but with a dramatically improved developer experience. Migrate to SwiftData when targeting iOS 17+.

## Failure Modes

1. **Main thread blocking** — executing a fetch request with 50,000 objects on the main context freezes the UI for 2–5 seconds → always perform large fetches on a background context (`container.newBackgroundContext()`) and merge results back to the main context. Use `NSFetchedResultsController` for table/collection view data — it batches fetches automatically.

2. **Migration failures in production** — adding a non-optional attribute without a default value to an existing entity causes the migration to fail and the app to crash on launch → always test migrations with your production data. Use `NSInferMappingModelAutomaticallyOption = true` for lightweight migrations. For complex migrations, write a custom mapping model. Ship a migration testing suite that loads the oldest supported model version and migrates through each version.

3. **Context concurrency violations** — accessing a managed object from a thread different from the context that created it crashes with "NSObjectInaccessibleException" → each `NSManagedObjectContext` is bound to a specific queue (`.mainQueueConcurrencyType` or `.privateQueueConcurrencyType`). Always use `context.perform { }` or `context.performAndWait { }` to interact with objects from the correct queue.

4. **Memory pressure from unfaulted objects** — fetching 100,000 objects without batching loads them all into memory, triggering a memory warning and app termination → set `fetchRequest.fetchBatchSize = 20` to fault objects in batches. Use `setPropertiesToFetch` to fetch only the columns you need. For count-only queries, use `countForFetchRequest` instead of fetching objects.

5. **Orphaned objects after relationship deletion** — deleting a parent object without configuring the delete rule leaves child objects pointing to a deleted parent → set appropriate delete rules: `.cascadeDeleteRule` (children deleted with parent), `.nullifyDeleteRule` (children's relationship set to nil), `.denyDeleteRule` (prevent deletion if children exist). The default is `.nullifyDeleteRule`.

6. **Merge conflicts between contexts** — a background context saves changes that conflict with unsaved changes in the main context, causing an `NSMergeConflict` → set the main context's `mergePolicy` to `NSMergeByPropertyObjectTrumpMergePolicy` (main context wins) or `NSMergeByPropertyStoreTrumpMergePolicy` (store wins). Listen to `NSManagedObjectContext.didSave` notifications and call `mainContext.mergeChanges(fromContextDidSave:)`.

7. **iCloud sync conflicts with CloudKit** — two devices modify the same record while offline. When both reconnect, Core Data's conflict resolution may silently choose the wrong version → use `NSPersistentCloudKitContainer` with a custom `NSMergePolicy`. Implement a `modifiedDate` attribute on entities and use `NSMergeByPropertyObjectTrumpMergePolicy` so the most recent change wins. Test offline conflict scenarios explicitly.

8. **Fetched results controller delegate crashes** — the `NSFetchedResultsControllerDelegate` methods update a table view, but the data source array and the table view are out of sync, causing an index-out-of-bounds crash → use `controller.performFetch()` inside `beginUpdates`/`endUpdates`. Ensure your data source is driven directly by `controller.fetchedObjects`, not a separate array.

9. **Persistent store file corruption** — the SQLite file becomes corrupted (e.g., due to a force-kill during a write), and subsequent launches crash → wrap store initialization in a `do/catch`. On failure, delete the corrupted store file and recreate it. Log the corruption event for diagnostics. Use WAL mode (`NSMigratePersistentStoresAutomaticallyOption`) for better crash resilience.

10. **Over-fetching with to-many relationships** — fetching a `User` entity with a to-many relationship to `Order` (10,000 orders) causes Core Data to fault and load all orders when navigating the relationship → use `setRelationshipKeyPathsForPrefetching` to prefetch only needed relationships. Alternatively, fetch `Order` entities directly with a predicate on `user.id` instead of navigating the relationship.

## Code Examples

### Standard Persistent Container Setup

```swift
import CoreData

final class PersistenceController {
    static let shared = PersistenceController()

    let container: NSPersistentContainer

    init(inMemory: Bool = false) {
        container = NSPersistentContainer(name: "AppModel")

        if inMemory {
            container.persistentStoreDescriptions.first?.url =
                URL(fileURLWithPath: "/dev/null")
        }

        container.loadPersistentStores { description, error in
            if let error = error {
                fatalError("Core Data store failed to load: \(error)")
            }
        }

        // Enable lightweight migration
        container.viewContext.automaticallyMergesChangesFromParent = true
        container.viewContext.mergePolicy = NSMergeByPropertyObjectTrumpMergePolicy
    }

    var viewContext: NSManagedObjectContext {
        container.viewContext
    }

    func backgroundContext() -> NSManagedObjectContext {
        container.newBackgroundContext()
    }

    func save() {
        let context = viewContext
        if context.hasChanges {
            do {
                try context.save()
            } catch {
                // In production, handle gracefully — do not fatalError
                print("Unresolved error \(error)")
            }
        }
    }
}
```

### Background Fetch + Main Context Merge

```swift
func fetchRecentOrders() async throws -> [Order] {
    let context = PersistenceController.shared.backgroundContext()

    return try await context.perform {
        let request: NSFetchRequest<Order> = Order.fetchRequest()
        request.sortDescriptors = [NSSortDescriptor(key: "createdAt", ascending: false)]
        request.fetchLimit = 50
        request.fetchBatchSize = 20

        do {
            return try context.fetch(request)
        } catch {
            throw CoreDataError.fetchFailed(underlying: error)
        }
    }
}

// Merge background changes into the main context
NotificationCenter.default.addObserver(
    forName: .NSManagedObjectContextDidSave,
    object: nil,
    queue: .main
) { notification in
    PersistenceController.shared.viewContext
        .mergeChanges(fromContextDidSave: notification)
}
```

### Batch Delete

```swift
func deleteExpiredSessions() throws {
    let context = PersistenceController.shared.container.newBackgroundContext()
    let cutoffDate = Calendar.current.date(byAdding: .day, value: -30, to: Date())!

    let request: NSFetchRequest<NSFetchRequestResult> = Session.fetchRequest()
    request.predicate = NSPredicate(format: "expiresAt < %@", cutoffDate as CVarArg)

    let deleteRequest = NSBatchDeleteRequest(fetchRequest: request)
    deleteRequest.resultType = .resultTypeObjectIDs

    try context.performAndWait {
        if let result = try context.execute(deleteRequest) as? NSBatchDeleteResult,
           let objectIDs = result.result as? [NSManagedObjectID] {
            // Merge deletion into main context
            NSManagedObjectContext.mergeChanges(
                fromRemoteContextSave: [NSDeletedObjectsKey: objectIDs],
                into: [PersistenceController.shared.viewContext]
            )
        }
    }
}
```

### Migration Testing

```swift
final class CoreDataMigrationTests: XCTestCase {
    func test_migrationFromV1ToCurrent() throws {
        // Load the V1 model
        let v1Model = NSManagedObjectModel.mergedModel(
            from: Bundle.main,
            forStoreConfiguration: nil,
            withVersionHash: "v1-hash"
        )!

        // Create a store with the V1 model
        let coordinator = NSPersistentStoreCoordinator(managedObjectModel: v1Model)
        let storeURL = FileManager.default.temporaryDirectory
            .appendingPathComponent("test-migration.sqlite")
        try coordinator.addPersistentStore(
            ofType: NSSQLiteStoreType,
            configurationName: nil,
            at: storeURL,
            options: nil
        )

        // Insert test data into V1
        let v1Context = NSManagedObjectContext(concurrencyType: .mainQueueConcurrencyType)
        v1Context.persistentStoreCoordinator = coordinator
        // ... insert objects ...
        try v1Context.save()

        // Now load the current model and attempt migration
        let currentModel = NSManagedObjectModel.mergedModel(from: [Bundle.main])!
        let newCoordinator = NSPersistentStoreCoordinator(managedObjectModel: currentModel)

        let options: [String: Any] = [
            NSMigratePersistentStoresAutomaticallyOption: true,
            NSInferMappingModelAutomaticallyOption: true
        ]

        // This will crash if migration fails
        try newCoordinator.addPersistentStore(
            ofType: NSSQLiteStoreType,
            configurationName: nil,
            at: storeURL,
            options: options
        )
    }
}
```

## Best Practices

- **Use `NSPersistentContainer` as your entry point.** It encapsulates the persistent store coordinator, managed object model, and default context setup. Avoid the older manual stack assembly unless you have specific requirements.
- **One context per thread.** The main context runs on `.mainQueueConcurrencyType`. Background work uses `.privateQueueConcurrencyType` via `container.newBackgroundContext()`. Never share a context across threads.
- **Enable lightweight migration in production.** Set `NSMigratePersistentStoresAutomaticallyOption = true` and `NSInferMappingModelAutomaticallyOption = true`. This handles adding/removing attributes, entities, and optional relationships automatically.
- **Test migrations with your oldest production schema.** Write a test that loads the V1 model, populates it with representative data, and migrates to the current model. This catches migration failures before users encounter them.
- **Use `NSFetchedResultsController` for table/collection views.** It batches fetches, monitors changes, and drives efficient UI updates with insert/delete animations. Do not fetch all objects into an array.
- **Set `fetchBatchSize` on fetch requests** that may return large result sets. A batch size of 20–100 is typical. This faults objects in batches, keeping memory usage predictable.
- **Use batch requests for bulk operations.** `NSBatchDeleteRequest` and `NSBatchUpdateRequest` operate directly on the SQLite store without loading objects. Always merge the results back into the main context.
- **Avoid `NSManagedObject` subclasses for read-only data.** If you are only reading data and never modifying it, use `NSFetchRequestResult` with `resultType = .dictionaryResultType` to get dictionaries instead of managed objects, reducing memory overhead.
- **Handle store corruption gracefully.** Wrap store loading in error handling. If the store fails to load, delete the corrupted file and recreate it. Log the event for telemetry.
- **Consider SwiftData for iOS 17+ projects.** It dramatically reduces boilerplate with `@Model` macros while using the same underlying Core Data stack. The migration path from Core Data to SwiftData is well-supported.
