# Edge Computing

## Description

Edge computing is a distributed computing paradigm that brings computation and data storage closer to the sources of data. Rather than sending all data to a centralized cloud for processing, edge computing processes data locally - at the "edge" of the network - near where it's generated. This approach reduces latency, bandwidth costs, and improves reliability.

The concept emerged from the need to handle the massive amounts of data generated by IoT devices, mobile applications, and autonomous systems. By processing data at the edge, organizations can:

- **Reduce latency**: Process data in milliseconds instead of round-trips to the cloud
- **Save bandwidth**: Only send relevant data to the cloud
- **Improve reliability**: Continue operating during network outages
- **Enhance privacy**: Keep sensitive data local
- **Scale efficiently**: Distribute processing across many edge nodes

Edge computing exists on a spectrum from simple CDN edge functions to sophisticated edge AI processing. The key principle is making decisions as close to the data source as possible, while only sending summaries or aggregated data to central systems.

Common edge computing scenarios include:
- IoT sensor data processing
- Video analytics at the edge
- AR/VR real-time processing
- Autonomous vehicle decision making
- Content delivery and personalization
- Industrial automation

## Purpose

**When edge computing is valuable:**
- For IoT applications with many sensors
- When low latency is critical
- In locations with limited connectivity
- For real-time analytics
- When processing large amounts of sensor data
- For privacy-sensitive applications

**When to avoid edge computing:**
- For simple applications with low data volumes
- When centralized processing is sufficient
- When infrastructure costs are a concern

## Rules

1. **Process locally when possible** - Only send data to cloud when needed
2. **Design for intermittent connectivity** - Edge nodes may lose connection
3. **Use appropriate compute at each tier** - Not everything needs full AI
4. **Implement sync strategies** - Handle data consistency
5. **Consider security at edge** - Protect data at each node
6. **Profile your workload** - Not all apps need edge
7. **Test in realistic conditions** - Network limits vary

### Edge IoT Processing

```python
# Edge device processing - Raspberry Pi
import time
from sensor import TemperatureSensor, HumiditySensor
from iotedge import EdgeModule, Message

class TemperatureMonitor(EdgeModule):
    def __init__(self):
        self.temp_sensor = TemperatureSensor()
        self.humidity_sensor = HumiditySensor()
        self.threshold = 30.0
        self.window_size = 60  # seconds
        self.readings = []
    
    def run(self):
        while True:
            temp = self.temp_sensor.read()
            humidity = self.humidity_sensor.read()
            timestamp = time.time()
            
            # Local processing - decide what to send
            reading = {'temp': temp, 'humidity': humidity, 'timestamp': timestamp}
            self.readings.append(reading)
            
            # Keep only recent readings
            cutoff = timestamp - self.window_size
            self.readings = [r for r in self.readings if r['timestamp'] > cutoff]
            
            # Only alert if threshold exceeded consistently
            if self.analyze_readings():
                self.send_alert()
            else:
                # Send aggregated data every 5 minutes
                if len(self.readings) > 0 and len(self.readings) % 300 == 0:
                    self.send_aggregate()
            
            time.sleep(1)
    
    def analyze_readings(self):
        """Check if temperature consistently above threshold"""
        high_readings = sum(1 for r in self.readings if r['temp'] > self.threshold)
        return high_readings / len(self.readings) > 0.8 if self.readings else False
    
    def send_alert(self):
        """Send immediate alert"""
        message = Message(
            body={'alert': 'high_temperature', 'readings': self.readings[-10:]},
            properties={'level': 'critical'}
        )
        self.send_to_output(message, 'alerts')
    
    def send_aggregate(self):
        """Send summary data"""
        avg_temp = sum(r['temp'] for r in self.readings) / len(self.readings)
        message = Message(
            body={
                'avg_temp': avg_temp,
                'min_temp': min(r['temp'] for r in self.readings),
                'max_temp': max(r['temp'] for r in self.readings),
                'count': len(self.readings)
            },
            properties={'level': 'normal'}
        )
        self.send_to_output(message, 'telemetry')
```

### Edge AI - TensorFlow Lite

```python
# Edge AI inference - TensorFlow Lite
import tflite_runtime.interpreter as tflite
import numpy as np
import cv2

class EdgeImageClassifier:
    def __init__(self, model_path='/models/classifier.tflite'):
        # Load model at edge
        self.interpreter = tflite.Interpreter(model_path)
        self.interpreter.allocate_tensors()
        
        # Get input/output details
        self.input_details = self.interpreter.get_input_details()
        self.output_details = self.interpreter.get_output_details()
    
    def preprocess(self, image):
        # Resize to model input
        image = cv2.resize(image, (224, 224))
        # Normalize
        image = image.astype(np.float32) / 255.0
        # Add batch dimension
        image = np.expand_dims(image, axis=0)
        return image
    
    def predict(self, image):
        # Preprocess
        input_data = self.preprocess(image)
        
        # Set input
        self.interpreter.set_tensor(
            self.input_details[0]['index'], 
            input_data
        )
        
        # Run inference
        self.interpreter.invoke()
        
        # Get output
        output_data = self.interpreter.get_tensor(
            self.output_details[0]['index']
        )
        
        # Post-process - return top prediction
        class_id = np.argmax(output_data[0])
        confidence = output_data[0][class_id]
        
        return {'class': class_id, 'confidence': float(confidence)}

# Use at edge
classifier = EdgeImageClassifier()
camera = cv2.VideoCapture(0)

while True:
    ret, frame = camera.read()
    if ret:
        result = classifier.predict(frame)
        
        if result['confidence'] > 0.8:
            # Only send alerts for high confidence
            send_to_cloud({
                'detection': result['class'],
                'confidence': result['confidence'],
                'timestamp': time.time()
            })
```

## Anti-Patterns

### 1. Over-processing at Edge

**Bad:**
- Running full ML models at edge when simpler rules work
- Processing data that should go directly to cloud

**Solution:**
- Profile workload
- Use simpler algorithms when possible
- Send raw data to cloud for complex processing

### 2. No Offline Handling

**Bad:**
- App fails when edge loses connection to cloud

**Solution:**
- Design for intermittent connectivity
- Cache critical data locally
- Queue operations for sync later

### 3. Not Securing Edge Devices

**Bad:**
- Edge devices with default passwords
- No encryption of local data
- Unpatched devices

**Solution:**
- Implement secure boot
- Encrypt data at rest
- Regular security updates
- Network segmentation

```python
# Handle sync between edge and cloud
class EdgeSync:
    def __init__(self, local_db, cloud_client):
        self.local = local_db
        self.cloud = cloud_client
        self.pending = []
    
    async def sync(self):
        # Push local changes
        for record in self.pending:
            try:
                await self.cloud.upload(record)
                self.pending.remove(record)
            except Exception as e:
                logger.warning(f"Sync failed: {e}")
        
        # Pull cloud changes
        changes = await self.cloud.get_changes(self.last_sync_time)
        for change in changes:
            self.local.apply(change)
        
        self.last_sync_time = time.time()
```

## Failure Modes

- **Edge node compromise exposing local data** → physical access to edge devices enables data extraction → sensitive data leakage → encrypt data at rest and minimize data stored at edge
- **Intermittent connectivity causing data loss** → edge node loses connection and buffer overflows → telemetry data permanently lost → implement persistent local storage with circular buffers and conflict resolution on reconnect
- **Inconsistent state between edge and cloud** → edge processes data differently than cloud logic → conflicting decisions → use identical processing logic at both tiers with feature flags for graceful degradation
- **Over-processing at resource-constrained edge** → running full ML models on low-power devices → battery drain and thermal throttling → use model quantization or simpler algorithms at edge
- **Stale edge firmware with known vulnerabilities** → edge devices not receiving security patches → botnet recruitment → implement OTA update mechanisms with automatic rollback on failure
- **Edge-to-cloud sync conflicts** → multiple edge nodes modify same resource → conflicting updates with no resolution → use CRDTs or last-write-wins with vector clocks for conflict resolution
- **No offline capability design** → edge application requires cloud connectivity → complete service outage during network partition → design core functionality to work offline with queued sync

## Key Takeaways

- Edge computing processes data near its source rather than in centralized data centers, reducing latency, bandwidth costs, and improving offline resilience
- Valuable for IoT applications, low-latency-critical systems, real-time analytics, privacy-sensitive applications, or locations with limited connectivity
- Avoid for simple applications with low data volumes, when centralized processing suffices, or infrastructure costs are prohibitive
- Tradeoff: reduced latency and bandwidth versus distributed complexity, device management overhead, and security challenges at physically accessible nodes
- Main failure mode: intermittent connectivity causing data loss when edge buffers overflow or sync conflicts arise between edge and cloud
- Best practice: design for offline operation, use tiered processing (device → edge cluster → cloud), encrypt data at rest, implement OTA updates, and use CRDTs for conflict resolution
