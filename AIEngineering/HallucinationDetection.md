# Hallucination Detection

## Description

Hallucination in AI refers to outputs that appear plausible but are factually incorrect, fabricated, or not supported by the model's training data. These false outputs can range from minor inaccuracies to completely invented facts, citations, or code that doesn't exist.

Hallucinations occur because LLMs are designed to generate statistically likely text, not necessarily truthful text. They predict what comes next based on patterns in training data, which can lead to confident-sounding but incorrect statements. Detecting and mitigating hallucinations is critical for building trustworthy AI systems.

The challenge is that LLMs often present false information with the same confidence as true information, making it difficult for users to distinguish fact from fiction. This is particularly dangerous in applications where accuracy is critical.

## Purpose

**When to focus on hallucination detection:**
- When AI outputs contain factual claims that users might act upon
- When generating code, documentation, or technical content
- When creating content that will be presented as authoritative
- In domains where accuracy has legal or safety implications
- When building AI systems that users trust for important decisions

**When to accept some risk:**
- Creative writing where fictional elements are acceptable
- Brainstorming where "wrong" ideas can still spark creativity
- When the cost of hallucination is low and detection cost is high
- Early exploration phases where accuracy isn't critical

## Rules

1. **Verify factual claims** - Cross-reference with trusted sources
2. **Check code references** - Ensure libraries, functions, and APIs actually exist
3. **Use confidence signals** - Monitor model uncertainty indicators
4. **Implement guardrails** - Block known hallucination patterns
5. **Combine methods** - No single technique catches all hallucinations
6. **User education** - Make users aware AI can hallucinate
7. **Continuous monitoring** - Track hallucination rates over time

## Examples

### Good Example: Factual Verification Pipeline

```python
class HallucinationDetector:
    def __init__(self):
        self.trusted_sources = TrustedSourceDatabase()
        self.fact_extractor = FactExtractor()
    
    async def verify_claims(self, text: str) -> VerificationResult:
        claims = self.fact_extractor.extract(text)
        
        verified_claims = []
        disputed_claims = []
        unverifiable_claims = []
        
        for claim in claims:
            source = await self.trusted_sources.search(claim)
            if source:
                if source.supports(claim):
                    verified_claims.append(claim)
                else:
                    disputed_claims.append(claim)
            else:
                unverifiable_claims.append(claim)
        
        return VerificationResult(
            verified=verified_claims,
            disputed=disputed_claims,
            unverifiable=unverifiable_claims
        )
```

### Bad Example: Trusting AI Output Without Verification

```
AI: "The Python function pandas.read_json() has a parameter called 
    'magic_mode' that automatically detects and parses nested JSON."

User: *uses this parameter in production code*
-> Crashes at runtime!
```

**Why it's bad:**
- The parameter doesn't exist
- No verification before using
- False information presented confidently

### Good Example: Code Existence Verification

```python
import ast
import importlib

def verify_code_hallucinations(code: str) -> CodeVerification:
    """Verify that imported libraries and functions exist."""
    
    errors = []
    warnings = []
    
    try:
        tree = ast.parse(code)
    except SyntaxError as e:
        errors.append(f"Syntax error: {e}")
        return CodeVerification(valid=False, errors=errors)
    
    # Check imports
    for node in ast.walk(tree):
        if isinstance(node, ast.Import):
            for alias in node.names:
                try:
                    importlib.import_module(alias.name)
                except ImportError:
                    errors.append(f"Module not found: {alias.name}")
        
        if isinstance(node, ast.ImportFrom):
            try:
                importlib.import_module(node.module)
            except ImportError:
                errors.append(f"Module not found: {node.module}")
    
    return CodeVerification(valid=len(errors) == 0, errors=errors, warnings=warnings)
```

### Bad Example: Accepting Fake Citations

```
AI: "According to Smith et al. (2023), 'quantum consciousness 
    has been proven through double-blind experiments at MIT.'

Real check: No such paper exists. MIT never conducted such research.
```

### Good Example: Confidence-Based Routing

```python
def handle_ai_response(response: str, confidence: float):
    """Route responses based on confidence level."""
    
    if confidence < 0.7:
        # Low confidence - requires human verification
        send_to_human_review(response)
        return "Your request requires verification. Please wait."
    
    elif confidence < 0.85:
        # Medium confidence - flag for optional review
        flag_for_review(response)
    
    # High confidence - proceed with standard validation

```

## Anti-Patterns

### 1. Over-Trusting High-Confidence Scores

**Bad:**
- Taking model confidence at face value
- Not distinguishing between factual and stylistic confidence
- Assuming high confidence = correct

**Why it's dangerous:**
- Models can be confidently wrong
- Confidence measures model certainty, not accuracy
- Calibration varies by domain

**Good:**
- Track actual accuracy at each confidence level
- Use confidence as one signal among many
- Implement domain-specific calibration

### 2. Relying on Single Verification Method

**Bad:**
- Only using fact-checking
- Only using code verification
- Only relying on human review

**Why it's bad:**
- Different hallucination types need different detection
- No method is 100% effective
- Gaps in coverage

**Good:**
- Layer multiple detection methods
- Combine automated and human checks
- Adapt to content type

### 3. Not Providing Source Attribution

**Bad:**
- Presenting AI output without citing sources
- Not linking claims to evidence
- Making it hard to verify

**Why it's bad:**
- Users can't verify claims
- No accountability for false information
- Spreads unverified knowledge

**Good:**
- Always cite sources in outputs
- Make verification easy
- Link to evidence

### 4. Ignoring Temporal Hallucinations

**Bad:**
- Not checking if information is outdated
- Presenting old information as current
- Not handling knowledge cutoffs

**Why it's bad:**
- AI training has cutoff dates
- Information changes over time
- Outdated info can be dangerous

**Good:**
- Date-stamp information
- Check for freshness
- Note knowledge cutoff in prompts

### 5. No Feedback Loop

**Bad:**
- Not tracking hallucinations that slip through
- Not learning from mistakes
- Repeating same errors

**Why it's bad:**
- Can't improve detection
- Same issues keep happening
- No accountability

**Good:**
- Log all hallucinations
- Analyze patterns
- Feed back into prompt engineering

## Failure Modes

- **Over-trusting high-confidence scores** → confidently wrong answers → dangerous decisions → calibrate confidence against actual accuracy per domain
- **Single verification method** → gaps in detection → hallucinations slip through → layer fact-checking, code verification, and human review
- **No source attribution** → unverifiable claims → spreading misinformation → always cite sources and link to evidence
- **Ignoring temporal hallucinations** → outdated info presented as current → wrong decisions → date-stamp outputs and check freshness
- **No feedback loop** → same hallucinations repeat → no improvement → log all hallucinations and feed patterns back into prompt engineering
- **Accepting fake citations** → invented papers/links → credibility loss → verify all references exist before publishing
- **Not verifying code references** → non-existent functions/libraries → runtime crashes → check imports and function existence before deployment

## Best Practices

### 1. Detection Techniques

**Self-Consistency Check:**
Ask the same question multiple ways and verify consistency.

**Cross-Reference Verification:**
Compare claims against multiple trusted sources.

**Confidence Calibration:**
Track actual accuracy vs. reported confidence.

**Semantic Similarity:**
Check if outputs match known facts semantically.

**Citation Verification:**
Verify that all cited sources and references exist.

### 2. Mitigation Strategies

**Prompt Engineering:**
- Ask for uncertainty acknowledgment
- Request sources with answers
- Use chain-of-thought to expose reasoning

**Retrieval-Augmented Generation (RAG):**
Ground outputs in verified document chunks.

**Fine-Tuning:**
Train on datasets with verified correct answers.

**Temperature Control:**
Use lower temperature for factual tasks.

### 3. User-Facing Safeguards

```python
def format_response_with_caveats(response: str, confidence: float):
    """Add appropriate hedging language based on confidence."""
    
    if confidence < 0.8:
        caveats = [
            "This information may not be accurate.",
            "Please verify this information independently.",
            "This is not professional advice."
        ]
        response += "\n\n⚠️ " + " ".join(caveats)
    
    return response
```

## Key Takeaways

- Hallucinations are plausible-sounding but factually incorrect outputs caused by LLMs predicting statistically likely text rather than truthful text
- Focus on detection when AI outputs contain factual claims, generate code/technical content, or operate in domains with legal/safety implications
- Accept some risk for creative writing, brainstorming, or early exploration where accuracy isn't critical
- Tradeoff: no single detection method catches all hallucinations; layering multiple techniques increases coverage but adds computational cost
- Main failure mode: over-trusting high-confidence scores—models can be confidently wrong, and confidence measures certainty not accuracy
- Best practice: combine self-consistency checks, cross-reference verification, citation validation, and confidence-based routing with feedback loops
- Related: AI validation, RAG, prompt engineering, fact-checking, input validation

## Additional Notes

**Types of Hallucinations:**

1. **Factual:** False claims about real-world facts
   - "Einstein won the Nobel for relativity" (he won for photoelectric effect)

2. **Semantic:** Wrong relationships between concepts
   - "Python is a compiled language"

3. **Logical:** Mathematically incorrect outputs
   - "2+2=5"

4. **Referential:** Invented citations, links, or resources
   - Citing papers that don't exist

5. **Code:** Non-existent functions, libraries, or APIs
   - Using made-up Python standard library functions

**Industry Benchmarks:**

- FActScore measures factual precision at sentence level
- TruthfulQA tests model's tendency to generate false answers
- HellaSwag tests common sense reasoning

**Known Limitations:**

- Detection is computationally expensive
- Some hallucinations are undetectable without external verification
- Verification sources can themselves be incomplete or wrong
